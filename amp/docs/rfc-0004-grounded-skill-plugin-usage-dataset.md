# RFC-0004: Grounded skill and plugin usage dataset

## Summary

Create a local, user-maintained usage and labeling dataset for Amp skills, plugin capabilities, AGENTS instructions, subagent prompts, and other agent-facing guidance.

The goal is to improve skills/plugins from grounded evidence in Chinh's real usage, not from static prompt vibes. A static “no-op instruction lint” is useful only as a weak signal. The durable habit should be:

```text
track real usage → label outcomes → monitor patterns → edit skills/plugins → measure again
```

This dataset becomes the source of truth for deciding whether an instruction is helpful, ignored, too broad, harmful, or a no-op.

For the dataset to be trustworthy, skills and plugins also need a version-controlled source of truth. Runtime files under `~/.config/amp` are not enough: usage labels must point to reviewable artifact versions, commits, and changes over time.

## Motivation

https://x.com/mattpocockuk/status/2069784839474032896

Agent-authored skills and plugin prompts tend to accumulate generic instructions such as “be thorough”, “write clearly”, “use best practices”, or “think carefully”. These lines often do not change agent behavior, but deleting them based only on intuition is still subjective.

The missing layer is an eval dataset generated from actual work:

- Which skills/plugins were relevant to a task?
- Which instructions were actually followed?
- Which instructions prevented mistakes?
- Which instructions were ignored?
- Which instructions produced worse behavior?
- Which lines appear meaningful but never correlate with better outcomes?

With labels maintained by Chinh, skill/plugin maintenance becomes evidence-based instead of aesthetic.

## Decision

Introduce a local dataset for skill/plugin usage evaluation.

The first version should be boring and manually correctable:

- append-only JSONL files under `~/.config/amp/logs/skill-plugin-usage/`
- one event row for each meaningful usage opportunity
- one or more label rows maintained by Chinh or by Amp with Chinh's confirmation
- version-controlled source artifacts under `~/Developer/agent-skills`, with `~/.config/amp` treated as the runtime projection
- reports generated by asking Amp to read the dataset, not by building a dashboard first
- skill/plugin edits should consult this dataset when the edit is about behavior, triggers, scope, or instruction quality

The dataset should optimize for being easy to inspect, diff, repair, and summarize.

## In scope

- Build a grounded dataset from real Amp usage.
- Track usage opportunities for skills, plugins, AGENTS instructions, capabilities, plugin prompts.
- Put skill/plugin source artifacts under version control so labels can point to exact versions.
- Let Chinh label outcomes with a small stable taxonomy.
- Support periodic monitoring.
- Avoid collecting secrets, raw transcripts, or excessive private context by default.

## Out of scope

- Do not build a dashboard in the first version.
- Do not build a fully automatic skill optimizer.
- Do not auto-delete or rewrite skills/plugins based on metrics alone.
- Do not treat static phrase matching as sufficient evidence of a no-op.
- Do not treat loose runtime files in `~/.config/amp` as the long-term source of truth for skills/plugins.
- Do not require every task to be labeled.
- Do not store full Amp transcripts by default.
- Do not compare agents by token efficiency; RFC-0002 already covers token usage as a separate ledger.

## Dataset location

Use a dedicated local directory:

```text
~/.config/amp/logs/skill-plugin-usage/
```

Initial files:

```text
events.jsonl
labels.jsonl
```

Optional future files:

```text
reports/YYYY-MM.md
snapshots/artifacts-YYYY-MM-DD.jsonl
```

Keep events and labels separate so labels can be added, corrected, or superseded without rewriting original usage rows.

## Version-controlled artifact source

Use `~/Developer/agent-skills` as the canonical version-controlled source for skills and Amp plugin artifacts. Treat `~/.config/amp` as a runtime install/projection location.

The exact layout can differ, but every evaluated artifact should have:

- a canonical repo path
- a runtime path, if installed into `~/.config/amp`
- a git commit or dirty working-tree marker at the time of use
- a documented sync/install path from repo to runtime

Direct edits under `~/.config/amp` should be treated as emergency/runtime edits. Durable changes should be ported back to `~/Developer/agent-skills` before they are considered complete.

Desired ownership model:

```diagram
╭──────────────────────────────╮       sync / install       ╭────────────────────╮
│ ~/Developer/agent-skills     │───────────────────────────▶│ ~/.config/amp      │
│ version-controlled source    │                            │ runtime projection │
╰──────────────┬───────────────╯                            ╰─────────┬──────────╯
               │ usage labels reference repo path + commit            │ Amp uses files
               ▼                                                      ▼
╭──────────────────────────────╮                            ╭────────────────────╮
│ skill-plugin-usage dataset   │◀───────────────────────────│ real Amp tasks     │
│ events + labels              │        outcomes            │ and plugin runs    │
╰──────────────────────────────╯                            ╰────────────────────╯
```

Suggested source layout:

```text
~/Developer/agent-skills/
  skills/<skill-name>/SKILL.md
  amp/plugins/<plugin-name>.ts
  amp/docs/tools/<capability>.md
  amp/AGENTS.md
  sync-skills.sh
```

This matters because the eval loop depends on comparing behavior across versions:

```text
label says instruction X was no-op → rewrite X in git → sync runtime → observe future labels
```

Without version control, the dataset can describe symptoms but cannot reliably explain which artifact change improved or regressed behavior.

## Usage event schema

Each event records a meaningful opportunity where an agent-facing artifact may have influenced behavior.

Example `events.jsonl` row:

```json
{
  "schema": "skill-plugin-usage-event/v1",
  "event_id": "evt_2026-06-28T12-34-56Z_a1b2c3",
  "captured_at": "2026-06-28T12:34:56.000Z",
  "thread_id": "T-019f0ec6-453f-70d9-9503-7d44d9f08135",
  "workspace": "/Users/lelouvincx/.config/amp",
  "task": {
    "user_intent": "Create a durable RFC for tracking and labeling real skill/plugin usage."
    "summary": "Created an RFC for a grounded skill/plugin usage dataset.",
    "tags": ["documentation", "rfc", "skills", "plugins"],
  },
  "artifacts": [
    {
      "type": "agents_instructions",
      "id": "global-agents",
      "source_repo": "/Users/lelouvincx/Developer/agent-skills",
      "source_path": "amp/AGENTS.md",
      "runtime_path": "/Users/lelouvincx/.config/amp/AGENTS.md",
      "vcs": {
        "commit": "abc1234",
        "dirty": false
      },
      "refs": ["AGENTS.md#Amp plugins"]
    },
    {
      "type": "rfc",
      "id": "RFC-0004",
      "source_repo": null,
      "source_path": null,
      "runtime_path": "/Users/lelouvincx/.config/amp/docs/rfc-0004-grounded-skill-plugin-usage-dataset.md",
      "vcs": null,
      "refs": []
    }
  ],
  "usage": {
    "trigger": "implicit",
    "phase": "documentation",
    "outcome": "completed"
  },
  "evidence": {
    "files_changed": ["docs/rfc-0004-grounded-skill-plugin-usage-dataset.md"],
    "tools_called": ["shell_command", "apply_patch"],
    "checks_run": [],
    "user_correction": "Static linting alone is not enough; need a grounded dataset from actual usage."
  },
  "privacy": {
    "contains_raw_transcript": false,
    "contains_file_contents": false,
    "contains_secrets": false
  }
}
```

Field notes:

- `event_id`: stable identifier for labels to reference.
- `thread_id`: Amp thread ID only; never store thread URLs or full thread links.
- `task.summary`: short enough to scan, detailed enough to understand the opportunity.
- `task.tags`: optional coarse tags for later filtering. Omit when tags are not obvious.
- `artifacts`: all relevant skills/plugins/instructions/docs that could have affected the work.
- `artifact.source_repo` and `artifact.source_path`: version-controlled source identity when available.
- `artifact.runtime_path`: installed file used by Amp during the task.
- `artifact.vcs`: commit and dirty-state metadata. Use `null` while the artifact has not yet been migrated into version control.
- `artifact.refs`: heading, section, instruction snippet hash, or other local pointer.
- `usage.trigger`: `explicit`, `implicit`, `automatic`, `agent_decision`, or `user_correction`.
- `usage.phase`: `planning`, `implementation`, `review`, `verification`, `documentation`, or `handoff`.
- `usage.outcome`: `completed`, `blocked`, `abandoned`, `partially_completed`, or `not_applicable`.
- `evidence.user_correction`: especially important because corrections are high-value labels.

## Label schema

Each label records Chinh's judgment about an event, artifact, or specific instruction.

Example `labels.jsonl` row:

```json
{
  "schema": "skill-plugin-usage-label/v1",
  "label_id": "lbl_2026-06-28T12-45-00Z_d4e5f6",
  "event_id": "evt_2026-06-28T12-34-56Z_a1b2c3",
  "labeled_at": "2026-06-28T12:45:00.000Z",
  "labeled_by": "chinh",
  "target": {
    "scope": "artifact",
    "type": "agents_instructions",
    "id": "global-agents",
    "source_repo": "/Users/lelouvincx/Developer/agent-skills",
    "source_path": "amp/AGENTS.md",
    "runtime_path": "/Users/lelouvincx/.config/amp/AGENTS.md",
    "ref": "AGENTS.md#Amp plugins"
  },
  "labels": ["helped", "correctly_triggered"],
  "verdict": "keep",
  "confidence": "medium",
  "notes": "Docs-first plugin rule shaped the RFC and avoided jumping directly to code."
}
```

Labels are append-only. If a label changes, append a new label row with the same `event_id` and target. The latest row wins for simple reporting, while history remains auditable.

## Label taxonomy

Start with a small taxonomy. Add new labels only when repeated real cases need them.

Positive labels:

- `helped`: the instruction/plugin improved the outcome.
- `correctly_triggered`: it activated in the right situation.
- `prevented_mistake`: it blocked an error, unsafe action, or wrong workflow.
- `improved_output_shape`: it made the response or artifact easier to use.
- `saved_time`: it reduced work or avoided repeated explanation.

Negative labels:

- `no_op`: no observable behavior difference from default agent behavior.
- `ignored_instruction`: the instruction was relevant but not followed.
- `wrong_trigger`: it activated when it should not have.
- `missed_trigger`: it should have activated but did not.
- `over_scoped`: it caused unnecessary work or broadened the task.
- `too_verbose`: it caused excessive output or context bloat.
- `tool_mismatch`: it pushed the wrong tool, skill, or plugin.
- `docs_code_drift`: docs and implementation disagreed.
- `unsafe_or_risky`: it encouraged behavior that violated safety or privacy expectations.
- `user_had_to_correct`: Chinh had to steer or correct the behavior.

Neutral labels:

- `not_applicable`: artifact was present but did not matter for this task.
- `unclear`: not enough evidence to judge.
- `needs_more_examples`: likely useful, but current instruction/example is under-specified.
- `needs_more_data`: wait for more traces before editing.

Verdicts:

- `keep`
- `rewrite`
- `delete`
- `split`
- `merge`
- `needs_more_data`

## No-op detection model

Static phrase matching should only create candidates. A line becomes a serious no-op candidate when usage evidence supports it.

Useful signals:

- The instruction is generic quality language.
- It appears in many relevant opportunities.
- It has no positive labels.
- It has `no_op`, `ignored_instruction`, or `user_had_to_correct` labels.
- Removing or ignoring it would not have changed observed outcomes.
- A more concrete neighboring instruction explains the actual behavior.

Example evaluation rule:

```text
If an instruction has at least 5 relevant usage opportunities,
0 positive labels,
and at least 2 negative/no-op labels,
mark it as a rewrite/delete candidate for Chinh's review.
```

This threshold is intentionally low and advisory. Chinh's label remains the ground truth.

## Monitoring questions

The dataset should make it easy to ask Amp questions like:

- Which skills/plugins helped me most this month?
- Which instructions did I correct most often?
- Which instructions look like no-ops based on actual usage?
- Which plugin docs drifted from implementation?
- Which tools were invoked but later judged unnecessary?
- Which skills triggered too broadly?
- Which tasks would benefit from better examples?
- Which artifact versions improved or regressed after a sync from `~/Developer/agent-skills`?
- Which runtime files in `~/.config/amp` lack a version-controlled source path?
- What should we delete, rewrite, or split next?

## Reporting shape

A useful report should be concise and action-oriented:

```text
Skill/plugin usage report — last 30 days

Most useful:
- AGENTS.md#Amp plugins: 4 helped labels, 0 negative labels.

Friction:
- docs/tools/<x>.md#Invocation: 2 missed_trigger labels.

Suspected no-ops:
- AGENTS.md#Version control: "write clear and concise commit messages" has 6 opportunities and no positive labels.

Recommended edits:
1. Rewrite <instruction> into a concrete output format rule.
2. Delete <instruction> after Chinh confirms.
3. Add an example to <skill/plugin> because labels show repeated ambiguity.

Version-control gaps:
- ~/.config/amp/plugins/<x>.ts has labels but no source path in ~/Developer/agent-skills.
```

Reports should show evidence and suggested changes, not silently modify artifacts.

## Capture strategy

Start automatic, but keep it predictable. The first capture mechanism should be a narrow canonical list of prefix-based phrases that cause a usage event to be recorded. Magic-word capture should be case-insensitive, prefix-based, and easy for Chinh to remember; broad natural-language intents should not be treated as capture triggers.

### Automatic first

Automatically capture a usage event when one of these happens:

- Chinh starts the incoming user message with a canonical capture prefix.
- Chinh starts the incoming user message with a canonical label prefix, which implies both event capture and label capture.
- Chinh runs the explicit `track event` command.
- Amp edits or reviews a tracked artifact: `AGENTS.md`, `SKILL.md`, `docs/tools/*.md`, plugin prompts, plugin docs, or subagent prompts.
- A tracked plugin capability, command, mode, or subagent wrapper is invoked.

Automatic capture should be conservative: at most one event per thread turn by default unless Chinh explicitly names multiple artifacts or labels.

The preferred detection point is turn start / incoming user message, before the assistant answers. `usage capture: skill ...` and `usage capture: plugin ...` capture the current or upcoming task. `usage label: ...` labels the previous meaningful agent turn by inspecting the current user message, the previous assistant response, the previous user request, and compact thread intent only when needed to resolve ambiguity. `usage report` triggers report behavior at turn start so the assistant can answer with the report in the same turn.

If Amp does not expose a turn-start or incoming-user-message plugin event, implement the same behavior as an agent-callable fallback whose description tells Amp to call it when the current user message starts with a canonical prefix. `agent.end` should not be the primary detection point; it may only be added later to enrich an already-captured pending event with compact outcome metadata.

### Canonical magic words

The only canonical capture prefixes are:

- `usage capture: skill <skill-or-instruction-target>`
- `usage capture: plugin <plugin-or-capability-target>`

Canonical label phrases trigger event capture plus a label when the target is clear. They must start with `usage label:` followed by one of these label phrases:

- `usage label: helped`
- `usage label: useful`
- `usage label: saved time`
- `usage label: no-op`
- `usage label: ignored instruction`
- `usage label: wrong trigger`
- `usage label: missed trigger`
- `usage label: wrong tool`
- `usage label: tool mismatch`
- `usage label: too verbose`
- `usage label: over-scoped`
- `usage label: docs/code drift`
- `usage label: unsafe`
- `usage label: risky`
- `usage label: had to correct`

The only canonical report phrase is:

- `usage report`

It should trigger report behavior at turn start, before the assistant answers, so the assistant can return the report in the same turn. It may also capture a usage event for the report request itself.

If a phrase maps clearly to a taxonomy label, append a label row using Chinh's words as the evidence note. If the artifact target is ambiguous, capture the event with `needs_more_data` or `unclear` rather than asking a broad follow-up question.

The canonical phrase list is itself a versioned artifact. New phrases should be added only when repeated real usage shows Chinh naturally uses them.

### Manual correction

Manual capture remains available for corrections, backfills, and edge cases, but those paths are not additional magic words. Prefer the canonical label prefix for simple previous-turn corrections:

```text
usage label: wrong tool
```

For corrections that need to supersede an earlier label, use the explicit `track event` / manual label workflow rather than adding extra free-form label commands:

```text
replacement label: helped; supersedes: <previous-label-id>
```

Because labels are append-only, corrections should append a superseding label row rather than editing historical rows.

### Explicit command palette capture

Add a command palette command for cases where Chinh wants to capture the current task without relying on magic words.

Suggested command:

```text
track event
```

The command should:

1. Capture the current Amp thread ID.
2. Use the latest user request and recent assistant outcome as the event seed when available.
3. Prompt for optional artifact target, labels, verdict, and notes.
4. Append an event row, and append a label row only when labels are provided.
5. Return the new `event_id` and any `label_id` values.

This command is the explicit fallback for “I know this should be tracked” moments. Magic words optimize for habit; the command palette optimizes for certainty.

## Capability implications

If implemented as Amp plugin capabilities, follow the existing docs-first plugin rule:

1. Add capability docs under `docs/tools/` first.
2. Make plugin code match the documented contract.
3. Keep events/labels append-only.

Likely future capabilities:

- `capture_skill_plugin_magic_words`: detect canonical prefixes at the start of incoming user messages and append event/label rows.
- `track_event`: command palette command for explicit capture of the current task.
- `log_skill_plugin_usage`: append an event row.
- `label_skill_plugin_usage`: append a label row.
- `skill_plugin_usage_report`: summarize recent evidence and suggested edits.

If Amp plugin events cannot observe user messages directly, implement the same behavior as an agent-callable tool whose description tells Amp to call it whenever the current user message starts with a canonical prefix. The intended product behavior is still automatic-first capture.

## Privacy and retention

Default rows must not include:

- plaintext secrets
- env values
- raw transcripts
- full user messages
- full file contents
- thread URLs or full thread links
- tool outputs that may contain private data

Rows may include:

- `thread_id`
- local file paths
- artifact IDs
- short task summaries
- manually written notes
- label judgments
- pointers to redacted audit logs

If richer evidence is needed, store a pointer to a redacted audit log rather than copying sensitive content into the dataset.

## Maintenance workflow

When editing a skill/plugin because of behavior quality:

1. Read recent relevant events and labels.
2. Identify whether the problem is trigger, scope, output shape, tool choice, safety, verbosity, or no-op language.
3. Edit the canonical artifact in `~/Developer/agent-skills`, not only the runtime copy in `~/.config/amp`.
4. Preserve or add a concrete behavior hook: trigger, source of truth, constraint, output shape, ordering rule, verification rule, or negative boundary.
5. Sync/install the updated artifact into `~/.config/amp`.
6. Record the source repo path and git commit/dirty state in the next usage event.
7. Add a follow-up event/label after the next real use.

The point is not to lint once. The point is to keep a living eval loop.

## Relationship to RFC-0002

RFC-0002 tracks raw token usage. This RFC tracks behavioral usefulness.

The two ledgers should stay separate:

- token usage answers “how much did agents cost?”
- skill/plugin usage answers “which guidance actually improved outcomes?”

Reports may join them later, but neither schema should depend on the other.

Completed:

- Capability docs exist for automatic prefix capture, the explicit `track event` command, and manual label correction.
- `plugins/skill-plugin-usage.ts` is active locally with:
  - `agent.start` capture for canonical incoming-message prefixes
  - `tool.result` capture for the first tracked subagent/plugin tools
  - the `track event` command-palette command
  - the `label_skill_plugin_usage` manual label/correction tool
  - local append-only JSONL writes and privacy-safe row shapes

Pause point:

Do not add more automation yet. Use the current capture loop for several real events and labels first. After enough real rows exist, return to this RFC to review the dataset and decide what to tune, delete, rewrite, or implement next.

Deferred until after several captures:

- Run the first usage report from real captured data.
- Tune canonical prefixes or label taxonomy only if real usage shows friction.
- Decide whether to track additional tools such as `send_to_thread` or command/agent-mode invocations.
- Use the first report to update one or two high-confidence instructions in version-controlled source.
- Add a manual/instruction for Chinh and save it in Logseq.

## Open questions

- Are the narrow canonical prefixes enough to capture Chinh's real usage without adding noisy natural-language triggers?
- How much task summary is enough without leaking too much context?
- Should instruction references use stable IDs, heading paths, snippet hashes, or all three?
- Should usage events reject artifacts without source repo metadata once migration is complete?
- What label volume is “good enough” for monthly review: 5, 10, or 20 labels?
- Should Chinh label only failures/corrections, or also strong positive examples?
- Should reports include token usage joins from RFC-0002, or keep behavior and cost separate until later?
